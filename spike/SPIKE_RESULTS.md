# KBO 데이터 스파이크 결과

> 목적: KBO 원팀 팬 앱을 설계하기 전에 필요한 데이터를 실제로 가져올 수 있는지만 판정하는
> 버리는 실험. 제품 아님. 실행일 기준 컨텍스트 날짜 = 2026-07-11(올스타 브레이크, 7/9~7/11 경기 없음).
> 환경: Python 3.13 + requests + beautifulsoup4 + lxml. 각 요청 사이 sleep, 개인·학습용.

## 한 줄 총평

**세 소스 중 뉴스(스파이크1)와 경기별 로그(스파이크3, 제일 중요)는 ✅ 완전히 됨 → 프로젝트 원안대로 가능.**
등록/말소(스파이크2)만 ⚠️ 부분적 — 공식 데이터 구조는 확인·파싱되지만 "당일"만 제공하고
날짜 이력·사유가 없어서 **앱 시작일부터 매일 수집(daily polling)** 방식이 필요하다.

| # | 소스 | 판정 | 핵심 |
|---|------|------|------|
| 1 | 네이버 스포츠 팀 뉴스 | ✅ 됨 | 팀별 JSON 엔드포인트. 헤드라인+URL+날짜 즉시 확보 |
| 2 | KBO 공식 등록/말소 | ⚠️ 부분적 | "당일" 등록/말소만(선수·포지션·팀). 날짜 이력·사유 없음 → 매일 수집 필요 |
| 3 | KBO 공식 경기별 타격 로그 | ✅ 됨 | 경기별 전 스탯 + 누적타율 시계열. 우리 계산이 공식 누적치와 정확히 일치 |

## 스파이크 1 — 네이버 스포츠 팀 뉴스 ✅ 됨

**질문:** 특정 팀 뉴스의 헤드라인+링크+날짜만 목록으로 가져올 수 있나? → **가능.**

**작동한 방법 / URL**
- 최신 `m.sports.naver.com`은 SPA라 HTML엔 내용이 없고, `api-gw.sports.naver.com`의 `/news/*`
  경로는 알 수 없는 라우트에 403을 준다(경로 추측 실패).
- **정답은 구(舊) 데스크톱 AJAX 엔드포인트**(현재도 살아있고 JSON 반환):
  ```
  https://sports.news.naver.com/kbaseball/news/list?isphoto=N&type=team&team={TEAM}&page=1
  ```
  - `type=team&team=LG` → 팀 한정 뉴스. 헤더에 `Referer: .../kbaseball/news/index`,
    `X-Requested-With: XMLHttpRequest` 권장.
  - 팀 코드: `LG, OB(두산), SS(삼성), LT(롯데), HT(KIA), SK(SSG), NC, WO(키움), HH(한화), KT`
- 응답 필드: `oid, aid, title, subContent, officeName, datetime, thumbnail, url, sectionName, totalCount`.
  기사 URL은 `url` 필드 직접 제공 or `https://n.news.naver.com/mnews/article/{oid}/{aid}`로 조립.
- 스크립트: `spike1_news.py` (본문 전문은 안 가져옴 — 헤드라인·링크·날짜만).

**샘플 데이터 (LG 트윈스, 실제 출력)**
```
 1. [2026.07.11 19:11] 마지막 잠실 올스타전 찾은 팬들…"잠실은 한 편의 청춘, 우리의 추억"  (뉴시스)
    https://n.news.naver.com/mnews/article/003/0014060726
 2. [2026.07.11 19:00] '잠실돔' 외야가 지금보다 좁아진다면? … 박해민의 '유쾌한 답' [백스톱]  (스포츠서울)
    https://n.news.naver.com/mnews/article/468/0001253033
 3. [2026.07.11 18:42] 야구는 졌지만, 이벤트는 이겼다…'구본혁-우강훈' LG, 삼성 꺾고 썸머 레이스 우승  (OSEN)
    https://n.news.naver.com/mnews/article/109/0005569798
 8. [2026.07.11 07:43] 타율 3할1푼→1할9푼8리 '대추락'…LG 4번타자 문보물이 깨어나야 한다  (OSEN)
    https://n.news.naver.com/mnews/article/109/0005569472
 ... (page=1에 9건, page 증가로 더 확보 가능)
```

**제약 / 메모**
- JSON이라 JS 렌더링·로그인 불필요. `page` 파라미터로 페이지네이션.
- `datetime`은 문자열(`YYYY.MM.DD HH:MM`). 정렬은 최신순.
- 신(新) api-gw 팀뉴스 라우트는 못 찾음 — 하지만 구 엔드포인트가 완벽히 동작하므로 불필요.

## 스파이크 2 — KBO 등록/말소(엔트리 변동) ⚠️ 부분적

**질문:** 누가 1군 등록/말소됐는지 목록을 가져올 수 있나? → **구조는 가능하나 "당일"만, 이력·사유 없음.**

**작동한 방법 / URL**
- `https://www.koreabaseball.com/Player/RegisterAll.aspx` (robots.txt 허용: `/ws/ /Member/ /Common/ /Help/`만 Disallow, 이 페이지는 허용)
- 페이지 하단에 **'당일 1군 등록'**, **'당일 1군 말소'** 두 위젯이 있고 컬럼은 `선수 | 포지션 | 팀`.
- 스크립트: `spike2_moves.py` — 두 표를 파싱해 `[등록/말소] 팀 선수(포지션) 날짜(=fetch일)` 출력.

**샘플 데이터 (실제 출력)**
```
fetch_date=2026-07-11  (KBO '당일' 위젯 = 실제 오늘)
1군 등록: 0건 / 1군 말소: 0건
컬럼: 선수 | 포지션 | 팀  (날짜=fetch일 암묵, 사유=원천에 없음)
(오늘은 이동 없음 — 2026 올스타 브레이크로 7/9~7/11 경기 없음)
```
> 오늘 0건인 이유는 스크래핑 실패가 아니라 **올스타 브레이크라 실제로 이동이 없어서**다.
> 경기가 있는 날엔 이 표가 `선수/포지션/팀`으로 채워진다(구조·파싱 검증 완료).

**막힌 지점 / 제약 (중요)**
- **날짜 이력 없음:** 이 위젯은 항상 "실제 오늘"만 보여준다. 페이지에 `hfSearchDate`(기본 `20260711`)
  숨은 필드가 있어 과거 날짜로 POST 백을 시도했지만, 그 값은 **로스터 스냅샷**에만 영향을 주고
  '당일 등록/말소' 위젯은 여전히 오늘 기준으로 렌더된다(과거 날짜도 "없습니다" 반환 확인).
  → 등록/말소는 **날짜 파라미터로 과거 조회가 안 됨.**
- **날짜 컬럼 없음:** 날짜는 "오늘"로 암묵. 수집 시 fetch 날짜를 붙여 기록해야 함.
- **사유(예: 슬럼프·부상) 필드: 원천에 존재하지 않음.** (예상대로 없음. 확인 완료 → 필요하면
  나중에 뉴스/LLM으로 생성해야 함.)
- STATIZ는 `statiz.co.kr`로 이전됐고 깔끔한 날짜별 엔트리 API가 없음. 구단들은 경기일마다
  SNS(트위터/페북/인스타)에 등록/말소를 게시(반정형·파싱 부담).

**→ 결론:** 등록/말소 피드는 **앱 런칭일부터 KBO '당일' 페이지를 매일 1회 폴링해 (fetch날짜 부착)
누적**하는 방식으로 만들어야 한다(스파이크3의 폴백과 같은 패턴). 과거 소급은 불가.

## 스파이크 3 — 경기별 로그 → 누적 타율/OPS 시계열 ✅ 됨 (제일 중요)

**질문:** 선수 경기별 기록을 모아 누적 타율/OPS 시계열을 만들 수 있나? → **완벽히 가능. 과거 소급까지 됨.**

**작동한 방법 / URL**
```
https://www.koreabaseball.com/Record/Player/HitterDetail/Daily.aspx?playerId={ID}
```
- 시즌 선택 `ddlYear`(기본 2026, 2019까지). `playerId`는 뉴스/검색으로 확보(예: 김도영=52605).
- 페이지는 **월별로 표 하나씩**(3월·4월·…), 각 데이터 행 = **경기 1개**:
  `날짜(MM.DD) | 상대 | AVG(그 경기) | PA | AB | R | H | 2B | 3B | HR | RBI | SB | CS | BB | HBP | SO | GDP | AVG(누적)`
- **맨 끝 `AVG(누적)` 컬럼이 이미 그 날짜까지의 누적 시즌 타율** → 우리 계산의 정답지로 교차검증.
- OPS는 우리가 계산: `OBP=(H+BB+HBP)/(AB+BB+HBP)`(SF 미노출로 소폭 근사), `SLG=TB/AB`
  (`TB=H+2B+2*3B+3*HR`), `OPS=OBP+SLG`.
- 스크립트: `spike3_timeseries.py` — 전 월 표를 평탄화해 (날짜→누적AVG, 누적OPS) 시계열 생성.

**샘플 데이터 (김도영 / KIA #5, 2026시즌, 실제 출력)**
```
player=김도영 (id 52605), season=2026, games=86
DATE          cumAVG  cumOPS   site_cumAVG(교차검증)
------------------------------------------------
2026-03-28     0.400   0.800   0.400
2026-03-29     0.250   0.650   0.250
2026-03-31     0.417   1.283   0.417
2026-04-01     0.357   1.143   0.357
2026-04-02     0.333   1.010   0.333
   ...
2026-07-04     0.297   1.010   0.297
2026-07-07     0.294   1.001   0.294
2026-07-08     0.295   1.000   0.295
2026-07-09     0.298   1.014   0.298
------------------------------------------------
season final: AVG=0.298 (site 0.298), OPS=1.014, AB=315 H=94 HR=27 BB=47
```
> **우리가 계산한 누적 타율이 공식 누적 컬럼과 매 시점 정확히 일치**(최종 0.298=0.298). 즉,
> "주식 그래프"의 원천 데이터(날짜→누적 타율/OPS 시계열)를 **과거 경기까지 통째로** 복원 가능.

**제약 / 메모**
- `playerId` ↔ 선수 매핑이 필요(뉴스 링크·선수검색으로 확보). 시즌은 `ddlYear`로 과거도 조회.
- SF(희생플라이)가 컬럼에 없어 OBP는 미세 근사(SLG·AVG는 정확). 필요하면 SF는 시즌 합계 페이지로 보정.
- 과거 로그가 **통째로 긁힘** → "앱 시작일부터 매일 수집" 폴백은 **불필요**. 스파이크1과 함께
  프로젝트 핵심 기능(선수 스탯 시계열 차트)이 데이터 측면에서 확정적으로 가능.

## 재현용 스크립트

| 파일 | 내용 |
|------|------|
| `spike1_news.py [팀코드]` | 네이버 팀 뉴스 10건(헤드라인+URL+날짜) |
| `spike2_moves.py` | KBO 당일 1군 등록/말소 파싱 |

```bash
cd kbo-spike && python3 -m venv .venv && ./.venv/bin/pip install requests beautifulsoup4 lxml pandas
./.venv/bin/python spike1_news.py LG
./.venv/bin/python spike2_moves.py
./.venv/bin/python spike3_timeseries.py 52605   # 김도영
```

## 프로젝트 함의 (기획 인계용)
1. **뉴스 피드·선수 스탯 차트는 지금 당장 가능**(안정적 JSON/HTML, 로그인·JS렌더 불필요).
2. **등록/말소는 "매일 폴링해 누적" 구조로 설계**해야 하고, **"사유"는 원천에 없으니 뉴스+LLM으로
   생성**하는 별도 파이프라인이 필요(원안의 예상과 일치).
3. 예의: 모든 요청에 sleep, robots.txt 준수, 개인·학습용. 데이터 권리는 KBO/네이버에 있으므로
   상업적 재배포 주의.
